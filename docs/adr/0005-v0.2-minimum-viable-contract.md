- **Status**: Accepted
- **Date**: 2026-04-19

## Context

ADR 0004 says: "v0.1 â†’ v1.0 implementation-first, do not freeze the schema, expect breaking changes between v0.x releases." That stance is correct for the engine's evolution.

But as of 2026-04-19 there is a conflict:

- `~/.claude/CLAUDE.md` Â§5 (DEC-018, 2026-04-18) has already promoted memory-hall to the **primary** memory write path for the seven-agent stack. The running server on `100.122.171.74:9100` is load-bearing.
- mem0 is now legacy (read-only). All new writes land in memory-hall.
- memory-gateway (the intended governance + auth wrapper per ADR 0003) has not yet been re-pointed from its mem0 proxy to memory-hall's library. Until that happens, memory-hall's HTTP surface itself is the production contract.
- Repo velocity is high (10+ commits on 2026-04-19 alone; v0.1 â†’ v0.2 in one day).

The risk: a v0.2.x patch that feels routine from the engine author's seat â€” rename a field, change an HTTP status, drop a column â€” silently breaks the seven-agent stack's writes. ADR 0004's "breaking changes are fine in 0.x" becomes a governance hole when "0.x" is already production.

This ADR narrows that hole without abandoning ADR 0004's freedom to iterate toward v1.0.

## Decision

**Define a Minimum Viable Contract (MVC) that v0.2.x patches must preserve. Everything outside the MVC remains free to change, as ADR 0004 allows.**

### In the v0.2 MVC (must not break during 0.2.x)

HTTP API â€” the three hot-path endpoints:

- `POST /v1/memory/write`
  - Request: `agent_id`, `namespace`, `type`, `content` are required; `tags`, `metadata`, `embedding_override`, `idempotency_key` are optional and may be ignored but must not cause 4xx.
  - Response 200: write succeeded and was indexed. Body contains `entry_id`, `content_hash`, `sync_status="synced"`.
  - Response 202: write was persisted but indexing is pending. Body contains `entry_id`, `content_hash`, `sync_status="pending"`.
  - Response 4xx: request invalid. The response body shape may evolve, but 4xx vs 2xx split is stable.
  - `content_hash` is deterministic for identical `(agent_id, namespace, type, content)` and returning the same `entry_id` for a duplicate write is part of the contract.

- `POST /v1/memory/search`
  - Request: `query` required; `mode` âˆˆ {`hybrid`, `semantic`, `lexical`}; `limit`, `namespace`, `agent_id`, `tags`, `type` optional filters.
  - Response: `{ total: int, hits: [...] }`. Each hit contains at minimum `entry_id`, `content`, `score`, `namespace`, `type`, `created_at`. Additional fields may appear; existing fields must not be renamed or removed.

- `GET /health`
  - Returns 200 with a JSON body when the engine is up. The exact shape of the body is **not** part of the MVC (it may grow); the 200 vs non-200 signal is.

Embedded runtime (critical for sandboxed agents â€” see README "Embedded use"):

- `from memory_hall import Settings, build_runtime`
- `runtime = build_runtime(settings=Settings())`
- `await runtime.start()` / `await runtime.stop()`
- `await runtime.write_entry(tenant_id, principal_id, payload=WriteMemoryRequest(...))`
- `await runtime.search_entries(tenant_id, payload=SearchMemoryRequest(...))`

These import paths and method signatures are frozen for v0.2.x. Internal implementation may change.

Data durability:

- SQLite file at `./data/memory-hall.sqlite3` is stable as the storage location.
- SQLite schema migrations within v0.2.x must be **forward-compatible**: no data loss, no column rename without a compatibility view, no destructive drop.
- A 0.2.x â†’ 0.2.(x+1) upgrade must not require manual re-indexing; if a reindex is needed it must run automatically on first start.

### Outside the v0.2 MVC (still free to change per ADR 0004)

- CLI command structure (`mh ...`) and flag names
- Config env var names
- Admin / reindex endpoints (`/v1/admin/*`, `/v1/memory/reindex*`)
- Observability / log line format
- Internal module boundaries inside `memory_hall/` (server, storage, embedder, cli) â€” callers must import from the top-level `memory_hall` only
- Dockerfile, docker-compose.yml, deploy layout
- Default port, default Ollama model, default embedder

### v0.3 gate

v0.3 is allowed to break anything above, but must ship:

1. A migration guide in `docs/migrations/v0.2-to-v0.3.md` before cutting the release tag.
2. At least one release-candidate tag (`v0.3.0-rc1`) with â‰¥48h soak before the final release.
3. An explicit note in the changelog listing every MVC item that changed.
4. No automatic upgrade of the production mini (`:9100`) â€” the production deployment is pinned to a specific minor version and bumped manually.

### Review date

This ADR is reviewed when memory-gateway is re-pointed at memory-hall (the moment the HTTP surface is no longer the public production contract for the seven-agent stack). At that review, the MVC may be narrowed further (since memory-gateway absorbs auth/governance) or widened (since fewer clients talk to memory-hall directly).

## Consequences

### Positive
- Patches within v0.2.x can ship without a week-long risk analysis â€” the rules are written down.
- Production memhall (`:9100`) has a written guarantee, not a handshake.
- ADR 0004's "iterate freely toward v1.0" is preserved: the MVC is narrow (three endpoints + one import path + storage durability), not the whole surface.
- memory-gateway's future `MemoryServiceProxy` replacement has a stable target to code against.

### Negative
- One more ADR to check before shipping a v0.2.x patch.
- Some small improvements (e.g. renaming a response field for clarity) get deferred to v0.3 even when they would be cheap in isolation.
- The MVC creates a psychological ceiling on how many breaking changes v0.2 can absorb; if too many accumulate as "planned for v0.3," v0.3 becomes a big-bang release, which is exactly the failure mode ADR 0004 wanted to avoid.

### Neutral
- The MVC is itself versioned with the ADR. A future ADR may narrow or widen it; it is not a permanent commitment.
- Minor documentation overhead: every v0.2.x release must explicitly confirm "MVC preserved" in the changelog.

## Alternatives Considered

| Option | Why rejected |
|---|---|
| Freeze the full v0.2 API (not just the MVC) | Contradicts ADR 0004 in substance; slows iteration on non-critical surface (admin endpoints, CLI) where change cost is low. |
| No freeze; rely on memory-gateway to arrive before any breakage | memory-gateway's timeline is not guaranteed (it already paused once in early 2026, per ADR 0003). Production cannot depend on an unfinished wrapper. |
| Freeze via semver only, document nothing | Semver alone doesn't say *which* contract is stable. Without the MVC list, "breaking change" is subjective. |
| Treat the production mini as a separate fork pinned indefinitely | Creates a long-lived branch divergence. The whole point of running memory-hall in production is to dogfood the mainline. |

## Operational implications

- Any PR that changes an MVC item must be labeled `breaking-v0.3` and rebased out of the v0.2.x release train.
- The `CHANGELOG.md` entry for every v0.2.x release must include the line: `MVC: preserved (see ADR 0005)`.
- The production mini's current pinned version is declared in `deploy/VERSION.md` (to be created in the next commit). Upgrading the mini is a deliberate act, not a side effect of `docker compose pull`.
- If an incident requires breaking the MVC to recover, it is allowed, but the incident report must file a matching retroactive ADR.
