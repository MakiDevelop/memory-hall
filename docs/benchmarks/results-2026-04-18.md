# v0.1 baseline — 2026-04-18

Frozen numbers from the day the engine shipped. Every later release should be measured against these. Don't edit this file; create `results-YYYY-MM-DD.md` per run.

## Environment

- Host: Mac mini M4, 32 GB, macOS
- Container: `memory-hall:0.1.0` (ARM64 Linux, multi-stage build)
- Embedder: Ollama `bge-m3` on DGX Spark via Tailscale (`OLLAMA_KEEP_ALIVE=-1`)
- Vector store: **SQLite brute-force fallback** (image's `sqlite-vec` wheel was ELFCLASS32 → vec0 didn't load; see v0.2 backlog)
- Storage: SQLite WAL + `busy_timeout=5000` + `BEGIN IMMEDIATE`
- Hall size at test time: 177 entries total (161 from concurrency stress tests, 16 real)

## Durability

| Test | Result |
|---|---|
| Dedup (write same content twice) | ✅ same entry_id, `created=false` on repeat |
| Round-trip (write → get by id, field-level compare) | ✅ all fields match |
| Persistence (`docker restart`) | ✅ 14/14 entries retained |
| Tenant isolation (vector-store layer) | ✅ cross-tenant search returns disjoint sets |
| Race (10 simultaneous identical writes) | ✅ 1 row, 1 `created=true` + 9 `created=false`, 0 errors |
| Ollama down (unreachable base URL) | ✅ write returns `202 Accepted` + `sync_status=pending`, row persisted, no 500 |

## Concurrency (writes)

See `concurrency.py`. Three scenarios, results:

| Scenario | entry_id success | sync_status | p50 / p95 / p99 latency | throughput |
|---|---|---|---|---|
| burst-10 | 10/10 (100%) | all pending | 11.1s / 19.3s / 20.0s | 0.49 write/s |
| burst-50 | 22/50 client-visible (44%) / server stored 23 | all pending | 23.3s / 42.4s / 44.1s | 0.49 write/s |
| sustained-100 over 30s | 87/100 client-visible / server stored 100 | all pending | 20.3s / 36.2s / 42.1s | 0.34 write/s |

**Key finding**: writes bottleneck on sequential Ollama embed calls. Server always persists rows even when client-side HTTP times out. No data loss under any scenario, but client-visible "success" < server-side "persisted".

## Recall — Hit@3 on 10 ground-truth pairs

See `hit3_recall.py` (edit `PAIRS` for your own data).

| Mode | Hit@3 |
|---|---|
| hybrid | **6/10 = 60%** |
| lexical | 6/10 = 60% |
| semantic (pure) | **0/10 = 0%** |

### Pattern

All 6 hits had at least one English or Latin-character token in the query. All 4 misses were pure-CJK queries relying on Chinese words being indexable as standalone tokens. Not a BM25 bug per se — root cause is that SQLite FTS5's `unicode61` tokenizer treats a continuous no-whitespace CJK run as a single token, so substring queries (e.g. query `撞牆` against indexed token `最近又撞牆`) miss.

Semantic-only mode failed on all 10 because short 3–8-char queries against a 177-entry corpus with ~90% noise don't discriminate well enough to put the target in top-3.

## Known caveats for interpreting these numbers

1. Hall had 161 entries from concurrency stress tests in `test:concurrency-*` namespaces. They pollute the vector space; real-world recall on a "clean" hall should be higher.
2. `sqlite-vec` extension didn't load in this image (ELFCLASS32 issue). All vector queries went through Python brute-force cosine. Scale limit is ~10k vectors before noticeable slowdown.
3. `/v1/memory?limit=500` returns HTTP 422 — list endpoint caps at `limit<=200`. Raise or document.
4. `HEALTHCHECK` occasionally reports `degraded` when the background reindex worker is saturating the embedder. Docker's health check then marks container unhealthy incorrectly; patch v0.2.

## v0.2 backlog from today's data

- **[recall]** Swap `unicode61` → jieba-based CJK tokenizer (Option D per Codex audit). Projected to raise Hit@3 on pure-CJK queries from 0% toward 80%.
- **[throughput]** Use `OllamaEmbedder.embed_batch()` (already implemented, currently unused) in the writer loop. Projected 5-10x batch speedup.
- **[ops]** Rebuild Docker image with 64-bit sqlite-vec wheel so vec0 virtual table engages. Brute-force scales to ~10k; vec0 comfortably to ~100k on commodity hardware.
- **[ops]** Health check should not compete with embedder work — use cached or cheap probe.
- **[ops]** List endpoint `limit` cap: either raise to 1000 or document the ceiling.
- **[test]** `tests/test_cjk_search.py` currently passes because semantic masks lexical failure — add a `score_breakdown.bm25 > 0` assertion for CJK phrases.
